üèóÔ∏è RDBMS Fundamentals

What is an RDBMS?

Relational Database Management System (RDBMS) is a software system that manages data using a structured format of tables (relations) with rows and columns. It supports SQL for querying and ensures data integrity, security, and consistency.

  • Data is stored in tables (relations).
  • Each row is a record; each column is a field/attribute.
  • Supports primary keys, foreign keys, and constraints.
  • Examples: MySQL, PostgreSQL, Oracle, SQL Server.
-- Example: A simple Employee table CREATE TABLE Employee ( emp_id INT PRIMARY KEY, name VARCHAR(50) NOT NULL, department VARCHAR(30), salary DECIMAL(10,2), hire_date DATE );

RDBMS vs DBMS

DBMS RDBMS
Stores data as files or collections Stores data in tables (relations)
No support for relationships Supports relationships via keys
No normalization Supports normalization
Examples: XML, File System Examples: MySQL, Oracle

Popular RDBMS Systems

MySQL
Open-source, widely used for web apps.
PostgreSQL
Advanced open-source, supports complex queries.
Oracle
Enterprise-grade, robust security and features.
SQL Server
Microsoft's RDBMS, integrates with .NET stack.

üìä Tables & Keys

What is a Table?

A table is a collection of related data organized in rows and columns. Each row is a record, and each column is a field.

-- Example: Student Table CREATE TABLE Student ( student_id INT PRIMARY KEY, name VARCHAR(50), age INT, email VARCHAR(100) );

Types of Keys

  • Primary Key: Uniquely identifies each row.
  • Foreign Key: Links to primary key in another table.
  • Candidate Key: Any column(s) that can be a primary key.
  • Composite Key: Combination of columns as primary key.
  • Alternate Key: Candidate keys not chosen as primary.
  • Super Key: Set of columns that uniquely identify a row.

üß† Practice Questions - Tables & Keys

Q1: What is a foreign key? Give an example. Easy
A foreign key is a column that creates a relationship between two tables. Example:
FOREIGN KEY (student_id) REFERENCES Student(student_id)
Q2: What is the difference between a primary key and a unique key? Medium
Primary key cannot have NULLs and only one per table.
Unique key can have one NULL and multiple unique keys per table.

üîß Normalization

What is Normalization?

Normalization is the process of organizing data to reduce redundancy and improve data integrity. It involves dividing tables and defining relationships.

  • 1NF: Eliminate repeating groups; atomic columns.
  • 2NF: Remove partial dependencies (non-key depends on part of composite key).
  • 3NF: Remove transitive dependencies (non-key depends on another non-key).
  • BCNF: Every determinant is a candidate key.
-- Example: Normalization Steps -- UNNORMALIZED TABLE Student_Courses +------------+----------------+-------------------------+ | student_id | student_name | courses | +------------+----------------+-------------------------+ | 1 | John Smith | Math, Physics, Chemistry| | 2 | Mary Johnson | Biology, Chemistry | +------------+----------------+-------------------------+ -- NORMALIZED TO 1NF Students Student_Courses +------------+----------+ +------------+----------+ | student_id | name | | student_id | course | +------------+----------+ +------------+----------+ | 1 | John | | 1 | Math | | 2 | Mary | | 1 | Physics | +------------+----------+ | 1 | Chemistry | | 2 | Biology | | 2 | Chemistry | +------------+----------+

2Ô∏è‚É£ Second Normal Form (2NF)

Rules:

  • Must be in 1NF
  • No partial dependencies on composite primary key
  • All non-key attributes fully depend on the entire primary key
-- VIOLATES 2NF (student_name depends only on student_id, not on full key) Student_Courses +------------+--------------+----------+-------+ | student_id | course_id | name | grade | +------------+--------------+----------+-------+ | 1 | CS101 | John | A | | 1 | CS102 | John | B | +------------+--------------+----------+-------+ Primary Key: (student_id, course_id) Problem: name depends only on student_id -- NORMALIZED TO 2NF Students Enrollments +------------+----------+ +------------+----------+-------+ | student_id | name | | student_id | course_id| grade | +------------+----------+ +------------+----------+-------+ | 1 | John | | 1 | CS101 | A | +------------+----------+ | 1 | CS102 | B | +------------+----------+-------+

3Ô∏è‚É£ Third Normal Form (3NF)

Rules:

  • Must be in 2NF
  • No transitive dependencies
  • Non-key attributes should not depend on other non-key attributes
-- VIOLATES 3NF (advisor_office depends on advisor_name, not student_id) Students +------------+----------+--------------+---------------+ | student_id | name | advisor_name | advisor_office| +------------+----------+--------------+---------------+ | 1 | John | Dr. Smith | Room 101 | | 2 | Mary | Dr. Smith | Room 101 | +------------+----------+--------------+---------------+ Problem: advisor_office depends on advisor_name (transitive dependency) -- NORMALIZED TO 3NF Students Advisors +------------+----------+ +--------------+---------------+ | student_id | name | | advisor_name | advisor_office| | advisor_id | advisor | +--------------+---------------+ +------------+----------+ | Dr. Smith | Room 101 | | 1 | John | +--------------+---------------+ | 2 | Mary | +------------+----------+

üß† Practice Questions - Normalization

Q1: What is 2NF? Give an example. Medium
2NF (Second Normal Form): A table is in 2NF if it is in 1NF and all non-key attributes are fully functionally dependent on the entire primary key.
Example: In a table with primary key (student_id, course_id), grade is fully dependent on both, but student_name only on student_id (violates 2NF).
Q2: What is the difference between 3NF and BCNF? When would you choose BCNF over 3NF? Hard
3NF: Every non-prime attribute is non-transitively dependent on every key.
BCNF: Every determinant is a candidate key.
Choose BCNF when there are anomalies in 3NF due to non-candidate key determinants.
Q3: Design a normalized database schema for an Online Bookstore. Consider books, authors, customers, orders, and reviews. Ensure it's in 3NF. Hard
Books, Authors, Customers, Orders, Order_Items, Reviews, Addresses tables, with foreign keys and no transitive dependencies. See example schema in the code above for details.

What is an ER Diagram?

An Entity-Relationship (ER) Diagram visually represents the structure of a database, showing entities (tables), their attributes (fields), and relationships between them. It helps in designing a robust database schema before implementation.

  • Entity: Real-world object (e.g., Student, Book)
  • Attribute: Property of an entity (e.g., student_id, name)
  • Relationship: Association between entities (e.g., Student borrows Book)
  • Cardinality: Defines number of instances (1:1, 1:N, M:N)
-- Example: Library ER Diagram (Textual) Entities: Student, Book, Loan Relationships: - Student borrows Book (via Loan) - Book is loaned to Student (via Loan) Student(student_id, name, ...) Book(book_id, title, ...) Loan(loan_id, student_id, book_id, loan_date, ...)

Relationships: Customer places Order, Order contains Product
Attributes:

‚Ä¢ Order: order_id (PK), order_date, customer_id (FK)
‚Ä¢ Product: product_id (PK), name, price

ER Diagram (Textual):

Q2: What is the difference between a strong and a weak entity? Give an example. Easy
Strong Entity: Has a primary key, exists independently (e.g., Student).
Weak Entity: Cannot exist without a related strong entity, uses a foreign key as part of its primary key (e.g., OrderItem depends on Order).
Example: Order (strong), OrderItem (weak, identified by order_id + item_id).

üíª SQL Fundamentals

What is SQL?

SQL (Structured Query Language) is the standard language for interacting with relational databases. It is used to create, modify, manage, and query data.

  • DDL: Data Definition Language (CREATE, ALTER, DROP)
  • DML: Data Manipulation Language (SELECT, INSERT, UPDATE, DELETE)
  • DCL: Data Control Language (GRANT, REVOKE)
  • TCL: Transaction Control Language (COMMIT, ROLLBACK)
-- Example: Basic SELECT Query SELECT name, salary FROM Employee WHERE department = 'HR';

üîó Joins & Relationships

What is a Join?

A JOIN combines rows from two or more tables based on a related column. It helps retrieve related data spread across multiple tables.

  • INNER JOIN: Returns rows with matching values in both tables.
  • LEFT JOIN: Returns all rows from the left table, and matched rows from the right table.
  • RIGHT JOIN: Returns all rows from the right table, and matched rows from the left table.
  • FULL OUTER JOIN: Returns all rows when there is a match in one of the tables.
  • CROSS JOIN: Returns the Cartesian product of both tables.
-- Example: INNER JOIN SELECT s.name, e.grade FROM Student s INNER JOIN Enrollments e ON s.student_id = e.student_id;

Subqueries & Nested Queries

A subquery is a query within another query. Useful for filtering, aggregation, and complex logic.

-- Example: Subquery SELECT name FROM Employee WHERE salary > (SELECT AVG(salary) FROM Employee);

Window Functions

Window functions perform calculations across a set of table rows related to the current row.

-- Example: ROW_NUMBER() SELECT name, salary, ROW_NUMBER() OVER (ORDER BY salary DESC) as rank FROM Employee;

Common Table Expressions (CTE)

-- Example: CTE WITH HighEarners AS ( SELECT name, salary FROM Employee WHERE salary > 70000 ) SELECT * FROM HighEarners;

üìà Indexes & Performance

What is an Index?

An index is a database object that improves the speed of data retrieval operations on a table at the cost of additional space and slower writes.

  • Speeds up SELECT queries
  • Can be created on one or more columns
  • Too many indexes can slow down INSERT/UPDATE/DELETE
-- Example: Creating an Index CREATE INDEX idx_salary ON Employee(salary);

üß† Practice Questions - Indexes

Q1: What are the pros and cons of using indexes? Medium
Pros: Faster SELECT queries.
Cons: Slower writes, more storage used.
Q2: When should you avoid using indexes? Medium
Avoid indexes on columns with many duplicate values or on small tables where scanning is faster.

üîí Transactions & ACID

What is a Transaction?

A transaction is a sequence of one or more SQL operations executed as a single unit. Transactions ensure data integrity and consistency.

  • Atomicity: All or nothing
  • Consistency: Valid state before and after
  • Isolation: Transactions do not interfere
  • Durability: Once committed, changes persist
-- Example: Transaction BEGIN; UPDATE Account SET balance = balance - 100 WHERE acc_id = 1; UPDATE Account SET balance = balance + 100 WHERE acc_id = 2; COMMIT;

üß† Practice Questions - Transactions & ACID

Atomicity, Consistency, Isolation, Durability
Q2: What is a dirty read? Medium
A dirty read occurs when a transaction reads data written by another transaction that has not yet been committed.
Q3: Write SQL to rollback a transaction if an error occurs. Medium
